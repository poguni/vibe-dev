Dodge It - 개발 구현 계획서 (Implementation Plan)

본 문서는 Research.md를 바탕으로 '똥 피하기 게임(Dodge It)'을 체계적으로 구현하기 위한 단계별(Phase) 개발 계획을 정리한 문서입니다. 프론트엔드(Next.js + Phaser 3), 백엔드(Supabase), AI(Gemini) 요소가 복합적으로 얽혀 있으므로, 순차적이고 독립적인 모듈 단위로 개발을 진행합니다.

🚀 Phase 1: 기반 아키텍처 및 환경 설정 (Foundation)

목표: 전체 애플리케이션이 구동될 뼈대를 잡고 핵심 라이브러리 연동 테스트를 완료합니다.
  1. 프로젝트 초기화
    • Next.js (App Router 기반) 프로젝트 생성 및 기본 구조 세팅.
    • 픽셀 아트/두들 스타일에 맞춘 전역 CSS (Tailwind CSS 또는 Vanilla CSS) 토큰 정의.
  2. Supabase 연동
    • Supabase 프로젝트 생성 및 초기 스키마 5개(users, problem_banks, problems, game_sessions, game_results) 구축.
    • 로컬 개발 환경과 Supabase 연결을 위한 환경 변수 세팅.
  3. Phaser 3 통합 테스트
    • Next.js의 컴포넌트 내에서 Phaser 3 캔버스가 정상적으로 마운트/언마운트되는지 래퍼(Wrapper) 컴포넌트 구현.
    • 테스트용 씬(Scene)에서 임의의 스프라이트(캐릭터)가 렌더링되는지 확인.
  4. Vercel 초기 배포
    • CI/CD 파이프라인 구축 및 Vercel 테스트 배포를 통한 환경 검증.

👩‍🏫 Phase 2: 교사 기능 및 문제 은행 구축 (Teacher & AI Features)

목표: 교사가 로그인하고, 문제를 생성 및 관리하는 백오피스 기능을 완성합니다.
  1. 교사 인증 시스템
    • Supabase Auth를 활용한 이메일/비밀번호 로그인 및 'Google로 계속하기' 구현.
  2. 수동 문제 업로드
    • 정해진 엑셀 양식을 파싱하여(예: xlsx 라이브러리 사용) problem_banks 및 problems 테이블에 일괄 Insert 하는 로직 구현.
  3. Gemini AI 문제 생성기 (핵심 기능)
    • 교과서 PDF 파일 업로드 및 텍스트 추출 로직 개발 (pdf-parse 또는 최신 OCR 서비스 연계).
    • 추출된 텍스트를 Gemini 1.5 Flash API로 전송 (15년 차 교사 페르소나 및 1~30개 단답형 문제 생성 프롬프트 엔지니어링).
    • AI가 반환한 JSON/텍스트 결과를 DB에 저장 및 교사가 확인/수정/삭제할 수 있는 UI 구현.

🚪 Phase 3: 게임 로비 및 실시간 세션 (Session & Real-time Lobby)

목표: 교사가 방을 만들고 학생이 접속하는 일련의 흐름을 실시간으로 구현합니다.
  1. 게임 방 생성
    • 교사 대시보드에서 특정 문제 은행을 선택해 세션 생성.
    • 유니크한 6자리 입장 코드 발급 및 QR코드(React-QR-Code 등 활용) 생성 UI 개발.
  2. 학생 간편 접속 (No-Login)
    • 학생용 랜딩 페이지(/join) 개발. 6자리 코드와 학번/이름 입력 폼.
  3. 실시간 대기실 (Supabase Realtime)
    • 학생이 접속할 때마다 Supabase 채널을 구독하여, 교사 화면(대기실) 및 학생 화면에 접속자 목록이 실시간(Realtime)으로 갱신되도록 연동.
    • 교사가 '게임 시작' 버튼을 누르면, 모든 접속된 학생 화면이 일제히 '카운트다운(3,2,1,Go)' 후 게임 화면으로 라우팅.

🎮 Phase 4: 메인 게임 엔진 및 물리 구현 (Game Logic)

목표: Phaser 3를 이용해 핵심적인 게임 메커니즘을 완성합니다. (이 단계가 가장 복잡하며 세부 단계로 나뉩니다)
  1. 게임 씬(Scene) 구성 및 에셋 로드
    • Preloader Scene: 캐릭터, 똥, 배경, 이벤트 이펙트 이미지 및 사운드 메모리 적재.
    • Main Game Scene: 게임 핵심 로직.
  2. 조작 및 UI 오버레이
    • 방향키(←, →)를 통한 하단 캐릭터 이동 물리엔진 적용 (월드 경계 충돌).
    • 게임 캔버스 위에 HTML DOM 요소를 오버레이(Absolute Positioning)하여 문제 출력 창과 정답 입력 <input> 창 배치.
    • [중요] 정답 입력창에 포커스가 가 있어도 방향키 입력은 무조건 캐릭터 이동으로만 바인딩되는 커스텀 키보드 이벤트 리스너(Input Event Intercept) 구현.
  3. 똥 낙하 및 충돌 시스템
    • 타이머 이벤트에 따라 화면 상단에서 랜덤하게 똥(장애물) 생성 (최대 5개 제한).
    • 난이도별(초기/중반/후반) 똥의 낙하 속도 증가 로직.
    • 바닥과 충돌 판정 (-50점), 캐릭터와 충돌 판정 (-100점, 하트 감소 로직).
  4. 문제 풀이 및 정답 판정 시스템
    • 정답 입력 시: 가장 Y축 좌표가 큰(바닥에 가까운) 똥 객체 1개를 파괴(Destroy)하고 애니메이션 재생. 점수 +100점, 다음 문제 로드.
  5. 특수 이벤트 시스템
    • 10번째 문제 통과 시 이벤트 모드 진입 판정.
    • 이벤트 정답 시 랜덤 함수를 돌려 물 내리기(모든 똥 객체 파괴 + 파란 물결 이펙트 + 500점) 또는 변비(Phaser 물리엔진 일시정지(physics.pause()) + 진동 트윈(Tween) 애니메이션) 발동.
  6. 게임 오버 처리
    • 하트가 0이 되거나 모든 문제를 풀면 게임 루프 정지, 결과를 DB(game_results)에 전송.

📊 Phase 5: 결과 시각화 및 통계 시스템 (Analytics & Polish)

목표: 데이터 시각화를 통해 사용자 경험을 높이고 프로젝트를 마무리합니다.
  1. 학생 결과 화면 (Podium)
    • 게임 종료 시 학생들에게 등수(1~3등)에 따라 단상(Podium)에 캐릭터가 올라가는 애니메이션 UI/CSS 연출.
    • 자신의 최종 점수와 학급 전체 리더보드 출력.
  2. 교사 대시보드 통계
    • 세션 종료 시 교사 화면에 학급 전체 결과 표 렌더링.
    • 차트 라이브러리(Recharts 등)를 도입하여 반 평균 점수 추이, 개별 학생 누적 데이터(학생 이름 클릭 시) 시각화 제공.
  3. 폴리싱 및 최적화
    • 전체 UI의 픽셀 아트 톤앤매너 검수.
    • BGM 및 효과음(정답, 오답, 피격, 물 내리기 등) 적용.
    • 다양한 크롬북 기기 해상도(16:9) 대응 뷰포트 최적화(Letterboxing 처리 등) 점검.
